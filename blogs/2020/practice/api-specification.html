    API设计规范

    Hyman 2020/3/2 实践总结规范

    # 全局规范

    为了保证API定义清晰易懂，风格统一，api统一遵循restfull风格。

    API接口url需要增加api前缀：http://example.com/api

    第一级Pattern为模块,比如：客户关系模块： /crm, 供应链管理：/scm

    第二级Pattern为资源分类

    # 版本

    为了能清晰的显示api版本信息，将版本信息定义在URL上，例如：http://example.com/api/v1

    # URL设计

    每个URL代表一种资源（resource），URL上资源使用名词复数形式，不能使用动词，多个单词用“-”隔开，统一用小写

    https://example.com/api/v1/crm/customers - 获取客户列表

    https://example.com/api/v1/crm/customers/1 - 获取id为1的客户信息

    避免多级 URL。在一些有父路径/子路径嵌套关系的资源数据模块中，路径可能有非常深的嵌套关系，例如:

    https://example.com/api/v1/crm/customers/{customer_id}/orders/{order_id}/goods/{good_id} - 获取某个客户下的某个订单的某个商品信息

    嵌套太深会导致语意不清晰，所以推荐在根(root)路径下指定资源来限制路径的嵌套深度（模块名必须）。使用嵌套指定范围的资源

    https://example.com/api/v1/crm/orders/{order_id} - 获取order_id的订单信息

    https://example.com/api/v1/crm/order/{order_id}/goods - 获取order_id的订单下的所有商品信息

    https://example.com/api/v1/crm/goods/{good_id}- 获取good_id信息

    # 资源操作

    对于资源的具体操作类型，由HTTP动词表示，只能使用：GET、POST、PUT、DELETE四种规范的请求方法：

    1. GET（SELECT）：从服务器取出资源（一项或多项）。
    2. POST（CREATE）：在服务器新建一个资源。
    3. PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）。
    4. DELETE（DELETE）：从服务器删除资源。

    例如：

    - GET /zoos：列出所有动物园
- POST /zoos：新建一个动物园
- GET /zoos/ID：获取某个指定动物园的信息
- PUT /zoos/ID：更新某个指定动物园的信息（提供该动物园的全部信息）
- DELETE /zoos/ID：删除某个动物园
- GET /zoos/ID/animals：列出某个指定动物园的所有动物
- DELETE /zoos/ID/animals/ID：删除某个指定动物园的指定动物

    # 请求

    • 在请求的body体统一采用JSON格式数据。

      在 PUT/PATCH/POST 请求的正文（request bodies）中使用JSON格式数据，而不是使用form 表单形式的数据。这与我们使用JSON格式返回请求相对应，例如:

      curl -H 'Content-Type: application/json' -X POST -d '{"name":"itart"}' http://example.com/api/v1/crm/customers
      1

      请求的url参数名统一格式为camelCase，不要使用snake_case，例如：GET /customers?firstName='test'，不是：GET /customers?first_name='test'

    • 结果过滤，排序，搜索，分页

      url最好越简短越好，对结果过滤、排序、搜索相关的功能都应该通过参数实现。

      过滤：例如你想限制GET /machines的返回结果：只返回哪些可用状态的machine， GET /machines?state=valid 这里的state就是过滤参数。

      排序：和过滤一样，一个好的排序参数应该能够描述排序规则，而不和业务相关。复杂的排序规则应该通过组合实现。排序参数通过 , 分隔，排序参数前加 - 表示降序排列。

      GET /tickets?sort=-createdTime #获取按创建时间降序排列票列表
GET /customers?sort=-createdTime #获取按创建时间降序排列的customers列表
      1
      2

    • 搜索：有些时候简单的排序是不够的。我们可以使用搜索技术来实现

      GET /machines?state=valid&sort=-priority,createAt # 获取优先级最高且打开状态的ticket，而且包含单词return的ticket列表
GET /machines?page=2&size=100 #指定第几页（page），以及每页的记录数（size）
      1
      2

    • 限制API返回值的域 有时候API使用者不需要所有的结果，在进行横向限制的同时（例如值返回API结果的前十个），还应该可以进行纵向限制，并且这个功能能有效的提高网络带宽使用率和速度。可以使用fields查询参数来限制返回的域例如：

      GET /machines?fields=id,ip,createdTime&state=valid&sort=-createdTime
      1

    # 状态码

    状态码必须精确

    客户端的每一次请求，服务器都必须给出回应。回应包括 HTTP 状态码和数据两部分。

    HTTP 状态码就是一个三位数，分成五个类别。

    # 1xx：相关信息
# 2xx：操作成功
# 3xx：重定向
# 4xx：客户端错误
# 5xx：服务器错误
